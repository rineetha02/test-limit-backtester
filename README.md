# test-limit-backtester

AEC-Q001 PAT limit-setting and backtest pipeline for semiconductor parametric test engineering. Demonstrates that spec-only screening misses mavericks: parts that pass the datasheet spec window but are reliability-escape risks.

> All bundled synthetic data is generated by the tool itself. Not from any real company, product, or fab.

---

## Two halves

**Python pipeline** — computes Cp/Cpk, proposes PAT limits, backtests against holdout lots, and generates Markdown/HTML reports.

**Web app** (`web/`) — a Next.js demo that loads pre-exported JSON from the pipeline and walks through the results in a guided flow. Runs entirely on Vercel with no server-side Python.

---

## Python pipeline

### Install

```bash
pip install -r requirements.txt
```

Requires: `pandas`, `numpy`, `scipy`, `matplotlib`. Python 3.9+.

### Run the synthetic demo

```bash
bash examples/run_demo.sh
```

Generates 10 lots x 1,000 parts of TMR current-sensor data, plants mavericks, runs the full pipeline, and writes:

- `reports/report.md`, `reports/report.html`
- `reports/figs/*.png` — per-parameter histograms

### Run with real datasets

```bash
python3 scripts/adapt_secom.py      # downloads UCI SECOM dataset (~1,567 parts)
python3 scripts/adapt_stsawfd.py    # downloads STMicro ST-AWFD D2 (~126,794 parts)

# Then run the pipeline pointing at those files:
python3 scripts/profile_tests.py  --data data/secom_data.csv --spec data/secom_spec.csv --output-dir reports/secom
python3 scripts/propose_limits.py --data data/secom_data.csv --spec data/secom_spec.csv --output-dir reports/secom
python3 scripts/backtest.py       --data data/secom_data.csv --limits reports/secom/proposed_limits.csv --spec data/secom_spec.csv --output-dir reports/secom
python3 scripts/make_report.py    --data data/secom_data.csv --backtest reports/secom/backtest_results.json --spec data/secom_spec.csv --limits reports/secom/proposed_limits.csv --capability reports/secom/capability_table.csv --output-dir reports/secom
```

### Use with your own data

Prepare two CSVs:

**`data.csv`** — one row per part:

| Column | Required | Description |
|--------|----------|-------------|
| `lot_id` | yes | Lot label |
| `lot_role` | yes | `reference` or `holdout` |
| `true_latent_fail` | no | 1 = known maverick (pass-spec, fail-PAT), else 0 |
| `spec_fail` | no | 1 = genuine spec failure, else 0 |
| `<param_1>` ... `<param_N>` | yes | One column per measurement |

**`spec.csv`** — one row per parameter:

| Column | Description |
|--------|-------------|
| `parameter` | Must match column name in data CSV |
| `unit` | Display unit |
| `lsl` | Lower spec limit (leave blank if none) |
| `usl` | Upper spec limit (leave blank if none) |
| `sided` | `two`, `upper`, or `lower` |

### Regenerate web data

After running the pipeline (or modifying outputs), regenerate the static JSON for the web app:

```bash
python3 scripts/export_web_data.py
```

This writes to `web/public/data/<dataset>/` — meta, capability, limits, histograms, backtest, and drift JSON files.

## Web app

### Run locally

```bash
cd web
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

### Deploy to Vercel

1. Push the repo to GitHub.
2. In [Vercel](https://vercel.com), add a new project and import the GitHub repo.
3. Set **Root Directory** to `web/`.
4. Accept the auto-detected Next.js build settings. No environment variables needed.
5. Deploy. The live URL renders all three datasets and the upload path.

The web app is fully static. It reads pre-exported JSON from `public/data/` and runs all client-side analysis in the browser. No server-side Python at runtime.

## Script reference

| Script | Inputs | Outputs | Purpose |
|--------|--------|---------|---------|
| `generate_synthetic_data.py` | none | `data/synthetic_test_data.csv`, `data/parameter_spec.csv` | 10 lots, 10k parts, planted mavericks and spec failures |
| `profile_tests.py` | data CSV, spec CSV | `reports/capability_table.csv` | Cp, Cpk (IQR/1.35 robust sigma) |
| `propose_limits.py` | data CSV, spec CSV | `reports/proposed_limits.csv` | PAT limits (robust median +/- N x robust_sigma), Cpk-target limits |
| `backtest.py` | data CSV, limits CSV, spec CSV | `reports/backtest_results.json`, `reports/backtest_summary.csv` | Confusion matrix, recall, DPPM, yield loss |
| `make_report.py` | all outputs | `reports/report.md`, `reports/report.html`, `reports/figs/*.png` | Engineer-ready report with histograms |
| `adapt_secom.py` | (downloads UCI SECOM) | `data/secom_data.csv`, `data/secom_spec.csv` | Adapter for SECOM real dataset |
| `adapt_stsawfd.py` | (downloads STMicro ST-AWFD D2) | `data/stsawfd_data.csv`, `data/stsawfd_spec.csv` | Adapter for ST-AWFD real dataset |
| `export_web_data.py` | all pipeline outputs | `web/public/data/<dataset>/*.json` | Export static JSON for the web app |

## Key domain invariants

- **Robust sigma = IQR / 1.35** — AEC-Q001 convention, resistant to outlier inflation
- **Cp** defined only for two-sided specs; None for one-sided
- **Cpk thresholds:** under 1.00 not_capable, under 1.33 marginal, under 1.67 capable, 1.67+ excellent
- **PAT catch zone:** PAT limits sit strictly inside spec. Mavericks live between PAT and spec.
- **`true_latent_fail` and `spec_fail` are separate columns.** Recall is computed only against `true_latent_fail`. Conflating them would give spec-only screening credit it does not deserve.
- **Drift-induced over-kill is a feature.** LOT10's 6.42% yield loss is static PAT detecting a process shift. The pipeline reports it per-lot and frames it as motivation for dynamic PAT.
